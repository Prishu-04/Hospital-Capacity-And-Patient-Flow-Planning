# Backend/ML — Hospital Capacity & Patient-Flow Data Profiler

## What's in this folder

| File | What it is |
|---|---|
| `Datacleaning.ipynb` | **Main deliverable.** The full, executed notebook — schema/missingness profiling, data-quality checks, the fixed cleaning pipeline, feature engineering, segment analysis, 5 charts, insight memo, data limitations, and a responsible-AI/risk-check section. Already run top-to-bottom; open it and you'll see every output without re-running anything. |
| `Datacleaning.py` | The same pipeline as a plain script, for automation/CLI use (`python Datacleaning.py`). |
| `dataset1_baseline_2025.csv` | Raw input: Jan–Apr 2025 daily department records. |
| `dataset2_comparison_2026.csv` | Raw input: Jan–Apr 2026 daily department records (same schema). |
| `cleaned_baseline_2025.csv` | Cleaned + feature-engineered output of the baseline file. |
| `cleaned_comparison_2026.csv` | Cleaned + feature-engineered output of the comparison file. |
| `combined_cleaned_hospital_data.csv` | Both years combined, with a `year` column, for year-over-year analysis. |
| `data_dictionary.csv` | Column name, type, description, unit, and valid range for every raw and derived column. |
| `insight_memo.md` | The 2025 vs 2026 headline numbers, segment notes, and data-quality notes, generated by the notebook. |
| `charts/` | The 5 charts from the notebook, saved as standalone PNGs (useful for dropping into the frontend or a slide deck). |
| `requirements.txt` | Python dependencies. |

## How to reproduce

```bash
pip install -r requirements.txt
jupyter nbconvert --to notebook --execute --inplace Datacleaning.ipynb
# or, for the script version:
python Datacleaning.py
```

Both raw CSVs must be in this same folder. The pipeline is deterministic — no
random sampling is used anywhere — so re-running it regenerates identical
output files.

## What changed vs. the original cleaning script

The original `Datacleaning.py`/`.ipynb` ran without errors but had several
issues that only show up once you check the *output* against the raw data
rather than just the absence of exceptions. They're documented in detail
inside the notebook (Section 6), in short:

1. Mixed date formats (ISO / slash / dash, from 4 different source systems)
   were silently mis-parsed or dropped by a plain `pd.to_datetime()` call.
2. The mostly-blank `notes` flag column was mode-imputed, which overwrote
   ~95% of clean rows with a fabricated `"capacity reconciliation needed"`
   flag.
3. Outlier clipping used a single dataset-wide IQR bound, which collapsed
   Emergency department's real `emergency_arrivals` signal (mean ~30/day)
   down to ~7.5 because it was compared against every other department's
   much lower baseline.
4. Impossible values (occupied beds > operational beds) weren't checked
   at all.
5. 2 of the "5 charts" never actually rendered, because the code checked for
   columns (`bed_occupancy_rate`, `average_wait_time_minutes`) that don't
   exist in this dataset — the `if` blocks were always `False` and failed
   silently.
6. No data dictionary, no segment (department/region) breakdown, no saved
   insight memo, and no feature engineering (occupancy rate, capacity
   status, etc.) despite these being part of the project brief.

All six are fixed in `Datacleaning.ipynb` / `Datacleaning.py`, and the fixes
are explained inline rather than just applied silently — so the *reasoning*
is part of the deliverable, not just the corrected numbers.
